\doxysubsection{Detailed Description}
Model for implementing a registration view. 

This model implements the logic for the registration of a new user. It can deal with traditional username/password registration, or registration of pre-\/identified users using federated login.

The model exposes four fields\+:
\begin{DoxyItemize}
\item Login\+Name\+Field\+: the login name (used as an identity for the \mbox{\hyperlink{classWt_1_1Auth_1_1Identity_a1d680fd0dbf7a12684a511d74081eb06}{Identity\+::\+Login\+Name}} provider) -- this can be an email if the \mbox{\hyperlink{classWt_1_1Auth_1_1AuthService}{Auth\+Service}} is configured to use email addresses as identity
\item Choose\+Password\+Field\+: the password
\item Repeat\+Password\+Field\+: the password (repeated)
\item Email\+Field\+: if an email address is to be asked (and is not used as identity).
\end{DoxyItemize}

The largest complexity is in the handling of third party identity providers, which is initiated with a call to \mbox{\hyperlink{classWt_1_1Auth_1_1RegistrationModel_a9ed6f669cc27fac0fdbacc1f3461b9d0}{register\+Identified()}}.

When a user is re-\/identified with the same identity, then the model may require that the (original) user confirms this new identity. The model indicates that this button should be made visible with \mbox{\hyperlink{classWt_1_1Auth_1_1RegistrationModel_aaf93eb12dcf6c254cac6546b92c8bea7}{is\+Confirm\+User\+Button\+Visible()}}, the action to take is determined by \mbox{\hyperlink{classWt_1_1Auth_1_1RegistrationModel_acfcbcb6150c4528d4a8de3dc63f50a94}{confirm\+Is\+Existing\+User()}}, and \mbox{\hyperlink{classWt_1_1Auth_1_1RegistrationModel_a78e673a4574c13bb7010b6693d58a501}{existing\+User\+Confirmed()}} ends this process by merging the new identity into the existing user.
