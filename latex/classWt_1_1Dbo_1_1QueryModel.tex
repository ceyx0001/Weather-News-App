\doxysubsection{Detailed Description}
\subsubsection*{template$<$class Result$>$\newline
class Wt\+::\+Dbo\+::\+Query\+Model$<$ Result $>$}

A Wt M\+VC Model to view/edit query results. 

The model fetches results from the query and presents the data in a table. It supports sorting the underlying S\+QL query using Query\+::order\+By().

The default implementation of \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_ab51eb60e26b58aa538964bf1a5a4bcd8}{data()}} converts Query results to model data using \mbox{\hyperlink{structWt_1_1Dbo_1_1query__result__traits_a9e28304a061c7ad52cb34e34fc410759}{query\+\_\+result\+\_\+traits$<$\+Result$>$\+::get\+Values()}}. You may define your own data presentation using the underlying {\ttfamily Result} by specializing \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_ab51eb60e26b58aa538964bf1a5a4bcd8}{data()}} and accessing data from \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_afaec30850c8094e9e23395de7de49f3d}{result\+Row()}}.

You may selectively add fields that you want to display using \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_a41e5e15eebba8157890b1372c4c41312}{add\+Column()}}, or you can also add all columns based on the query using \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_afa1bbf12d2b58ae91481f6d09c1dd8db}{add\+All\+Fields\+As\+Columns()}}.

The model supports editing of the underlying data (even if the underlying query fetches results from multiple tables!). Values in columns that correspond to fields that have been mapped (and are writable) in a \mbox{\hyperlink{classDatabase}{Database}} Object can be edited. The default implementation of \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_ae1575222c8b1da8ea910d19190290a4c}{set\+Data()}} uses \mbox{\hyperlink{structWt_1_1Dbo_1_1query__result__traits_afd928ae3f5216574125647063135cc3a}{query\+\_\+result\+\_\+traits$<$\+Result$>$\+::set\+Value()}} to manipulate the database object, and thus uses the same write-\/behind properties as \mbox{\hyperlink{classWt_1_1Dbo_1_1ptr_aed605599617e3ae02d9e92b1daafa8e7}{ptr$<$\+C$>$\+::modify()}}. To customize editing, you can specialize \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_ae1575222c8b1da8ea910d19190290a4c}{set\+Data()}} and use \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_afaec30850c8094e9e23395de7de49f3d}{result\+Row()}} to modify the result object directly.

The model supports also inserting rows (only at the end), and removing rows, which are reflected in object additions and removals from the \mbox{\hyperlink{classWt_1_1Dbo_1_1Session}{Session}}.

Editing is directly to the underlying database objects (change, insert and remove). \mbox{\hyperlink{classNote}{Note}} that these changes will be flushed to the database whenever a transaction is committed, or before a query is run. The model will not explicitly create a transaction for the modification, but since the model uses a query for reading data, the change may be committed to the database depending on how the model is loading data. Still, this implies that usually inserting a row and setting its data happens within a single S\+QL {\ttfamily \char`\"{}insert\char`\"{}} statement.

To get good performance, the model keeps the following data cached\+:
\begin{DoxyItemize}
\item \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_abad12ccb058ef3ecd3c0db9d5f56783b}{row\+Count()}}
\item a batch of data, controlled by \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_a6871c2e7be184bfd2339320812387f81}{set\+Batch\+Size()}}
\end{DoxyItemize}

Moreover, the model will try to fetch a batch of data even if you ask for the \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_abad12ccb058ef3ecd3c0db9d5f56783b}{row\+Count()}}, which may provide the row count as a side-\/effect if the query returns less results than a single batch size. If you do not want this behaviour (which might be the case if you are only interested in \mbox{\hyperlink{classWt_1_1Dbo_1_1QueryModel_abad12ccb058ef3ecd3c0db9d5f56783b}{row\+Count()}}, not the actual data) then you can avoid this behaviour by setting batch\+Size to 0.
