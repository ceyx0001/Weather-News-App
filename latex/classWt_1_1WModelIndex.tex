\doxysubsection{Detailed Description}
A value class that describes an index to an item in a data model. 

Indexes are used to indicate a particular item in a \mbox{\hyperlink{classWt_1_1WAbstractItemModel}{W\+Abstract\+Item\+Model}}. An index points to the item by identifying its row and column location within a parent model index.

An index is immutable.

The default constructor creates an {\itshape invalid index}, which by convention indicates the parent of top level indexes. Thus, a model that specifies only a list or table of data (but no hierarchical data) would have as valid indexes only indexes that specify the {\itshape invalid} model index as parent.

Upon the model\textquotesingle{}s choice, model indexes for hierarchical models may have an internal Id represented by a int64\+\_\+t (\mbox{\hyperlink{classWt_1_1WModelIndex_aa1373ac26110e15238bd8379fdb04c3b}{internal\+Id()}}), a pointer (\mbox{\hyperlink{classWt_1_1WModelIndex_a870e6637b7beebef4f4dfaec9f08e0a2}{internal\+Pointer()}}).

Indexes are created by the model, within the protected \mbox{\hyperlink{classWt_1_1WAbstractItemModel_af9b04fdec198dfbc1c9297ead93c8930}{W\+Abstract\+Item\+Model\+::create\+Index()}} methods. In this way, models can define an internal pointer or id suitable for identifying parent items in the model.

When a model\textquotesingle{}s geometry changes due to row or column insertions or removals, you may need to update your indexes, as otherwise they may no longer point to the same item (but instead still to the same row/column). Thus, if you store indexes and want to support model changes such as row or columns insertions/removals, then you need to react to the corresponding signals such as \mbox{\hyperlink{classWt_1_1WAbstractItemModel_a54c971088bb9b424f97fe31c17f8f12f}{W\+Abstract\+Item\+Model\+::rows\+Inserted()}} to update these indexes (i.\+e. shift them), or even remove them when the corresponding row/column has been removed.

When a model\textquotesingle{}s layout changes (it is rearranging its contents for example in response to a sort operation), a similar problem arises. Some models support tracking of indexes over layout changes, using {\itshape raw} indexes. In reaction to \mbox{\hyperlink{classWt_1_1WAbstractItemModel_ab377ac2fcb9de9d191bd1353665d1389}{W\+Abstract\+Item\+Model\+::layout\+About\+To\+Be\+Changed()}}, you should encode any index which you wish to recover after the layout change using \mbox{\hyperlink{classWt_1_1WModelIndex_a93c11e136a0fdc596eb745cc7fce7033}{encode\+As\+Raw\+Index()}}, and in \mbox{\hyperlink{classWt_1_1WAbstractItemModel_afe7cb1d5186881f83aa8f0e452aaef66}{W\+Abstract\+Item\+Model\+::layout\+Changed()}} you can obtain an index that points to the same item using \mbox{\hyperlink{classWt_1_1WModelIndex_a068d2232b5ed7d2dcd9560305f82c0fe}{decode\+From\+Raw\+Index()}}.

Comparison operator. 

Returns {\ttfamily true} if the index comes topologically before {\ttfamily other}.

Topological order follows the order in which the indexes would be displayed in a tree table view, from top to bottom followed by left to right.

An invalid index comes before all other indexes. Indexes \mbox{\hyperlink{classWt_1_1WModelIndex_a93c11e136a0fdc596eb745cc7fce7033}{encoded as raw index}} come after the invalid index and before all other indexes, and are ordered according to their \mbox{\hyperlink{classWt_1_1WModelIndex_aa1373ac26110e15238bd8379fdb04c3b}{internal\+Id()}}. \mbox{\Hypertarget{classWt_1_1WModelIndex_a930100eea844de84e91bfecd0a87f601}\label{classWt_1_1WModelIndex_a930100eea844de84e91bfecd0a87f601}}
