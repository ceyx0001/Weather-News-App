\doxysubsection{Detailed Description}
A base class for objects that participate in the signal/slot system. 

The main feature offered by W\+Object is automated object life-\/time tracking when involved in signal/slot connections. Connections between signals and slots of W\+Object instances implement a type-\/safe event callback system. For example, one can simply connect() the \mbox{\hyperlink{classWt_1_1WInteractWidget_af6cb297f8593e6884de66d23ea69d49c}{W\+Interact\+Widget\+::clicked()}} signal of a \mbox{\hyperlink{classWt_1_1WPushButton}{W\+Push\+Button}} to the \mbox{\hyperlink{classWt_1_1WApplication_a5231d54ed34982f4366058eb6440c8f7}{W\+Application\+::quit()}} method, to exit the application when the button is clicked\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{std::unique\_ptr<Wt::WInteractWidget> sender\{\textcolor{keyword}{new} \mbox{\hyperlink{classWt_1_1WText}{Wt::WText}}(\textcolor{stringliteral}{"Quit."})\};}
\DoxyCodeLine{\mbox{\hyperlink{classWt_1_1WApplication}{Wt::WApplication}} *app = \mbox{\hyperlink{classWt_1_1WApplication_a27690e10fb1dffef9b688b5a6ecf26b0}{Wt::WApplication::instance}}();}
\DoxyCodeLine{sender-\/>clicked().connect(app, \&\mbox{\hyperlink{classWt_1_1WApplication_a5231d54ed34982f4366058eb6440c8f7}{Wt::WApplication::quit}});}
\end{DoxyCode}


Wt\textquotesingle{}s signals may also propagate arguments to slots. For example, the same clicked() signal provides event details in a \mbox{\hyperlink{classWt_1_1WMouseEvent}{W\+Mouse\+Event}} details class, and these details may be received in the slot\+:

In conjunction with \mbox{\hyperlink{classWt_1_1EventSignal}{Event\+Signal}}, W\+Object also facilitates learning of client-\/side event handling (in Java\+Script) through invocation of the slot method. This is only possible when the slot behaviour is stateless, i.\+e. independent of any application state, and can be specified using the \mbox{\hyperlink{classWt_1_1WObject_adaa163b9e92933f3b2ff4ec58e2734c6}{implement\+Stateless()}} methods.

On-\/demand stateless slot implementation. 

This method returns a stateless slot implementation for the given {\ttfamily method}. To avoid the cost of declaring methods to be stateless when they are not used, you may reimplement this method to provide a stateless implementation for a method only when the method is involved in a slot connection.

Use \mbox{\hyperlink{classWt_1_1WObject_adaa163b9e92933f3b2ff4ec58e2734c6}{implement\+Stateless()}} to provide a stateless implementation of the given {\ttfamily method}, or return the base class implementation otherwise.

Returns the (unique) identifier for this object. 

For a W\+Widget, this corresponds to the id of the D\+OM element that represents the widget. This is not entirely unique, since a \mbox{\hyperlink{classWt_1_1WCompositeWidget}{composite widget}} shares the same id as its implementation.

By default, the id is auto-\/generated, unless a custom id is set for a widget using \mbox{\hyperlink{classWt_1_1WWidget_a93ff9d1ca04e733cd54cd40a06775f7e}{W\+Widget\+::set\+Id()}}.

This method sets the Java\+Script implementation for a method. As a result, if Java\+Script is available, the Java\+Script version will be used on the client side and the visual effect of the C++ implementation will be ignored.

This is very similar to an auto-\/learned stateless slot, but here the learning is avoided by directly setting the Java\+Script implementation.

The {\ttfamily js\+Code} should be one or more valid Java\+Script statements.

Declares a slot to be stateless and learn client-\/side behaviour on first invocation. 

Indicate that the given slot is stateless, and meets the requirement that the slot\textquotesingle{}s code does not depend on any state of the object, but performs the same visual effect regardless of any state, or at least until \mbox{\hyperlink{classWt_1_1WObject_a5c7dcd5a60c88e5b36f0c8e0f9117698}{reset\+Learned\+Slot()}} is called.

When this slot is connected to an \mbox{\hyperlink{classWt_1_1EventSignal}{Event\+Signal}} (such as those exposed by \mbox{\hyperlink{classWt_1_1WInteractWidget}{W\+Interact\+Widget}} and \mbox{\hyperlink{classWt_1_1WFormWidget}{W\+Form\+Widget}}), the Wt library may decide to cache the visual effect of this slot in Java\+Script code at client-\/side\+: this effect will be learned automatically at the first invocation. This has no consequences for the normal event handling, since the slot implementation is still executed in response to any event notification. Therefore, it is merely an optimization of the latency for the visual effect, but it does not change the behaviour of the application.

When for some reason the visual effect does change, one may use \mbox{\hyperlink{classWt_1_1WObject_a5c7dcd5a60c88e5b36f0c8e0f9117698}{reset\+Learned\+Slot()}} or \mbox{\hyperlink{classWt_1_1WObject_a92d8b00edc79b7a60d41d7c088f50436}{reset\+Learned\+Slots()}} to flush the existing cached visual effect, forcing the library to relearn it.

It is crucial that this function be applied first to a slot that is intended to be stateless before any Event\+Signal connects to that slot. Otherwise, the connecting Event\+Signal cannot find the stateless slot implementation for the intended slot, and the statement will have no effect for that connection.
