\doxysubsection{Detailed Description}
Represents an application instance for a single session. 

The recipe for a Wt web application, which allocates new \mbox{\hyperlink{classWt_1_1WApplication}{W\+Application}} instances for every user visiting the application is thus\+:

The W\+Application object provides access to session-\/wide settings, including\+:


\begin{DoxyItemize}
\item circumstantial information through \mbox{\hyperlink{classWt_1_1WApplication_ad51548b74173d6ec294cbf7ef31e8e6f}{environment()}}, which gives details about the user, start-\/up arguments, and user agent capabilities.
\item the application title with \mbox{\hyperlink{classWt_1_1WApplication_a71a3f7da5abb9a76df94fab69ba61670}{set\+Title()}}.
\item inline and external style sheets using \mbox{\hyperlink{classWt_1_1WApplication_a6a9a20d65ce9e7c2f62b27387c94e10d}{style\+Sheet()}} and \mbox{\hyperlink{classWt_1_1WApplication_a8dcc025cd58968e087de8e90e6a4f547}{use\+Style\+Sheet()}}.
\item inline and external Java\+Script using \mbox{\hyperlink{classWt_1_1WApplication_a2a92457b9212cef4057cb54e56183967}{do\+Java\+Script()}} and \mbox{\hyperlink{classWt_1_1WApplication_ae88546224ed32cade783da188b145138}{require()}}.
\item the top-\/level widget in \mbox{\hyperlink{classWt_1_1WApplication_a485af3ad2b758931edbde77fcf529981}{root()}}, representing the entire browser window, or multiple top-\/level widgets using \mbox{\hyperlink{classWt_1_1WApplication_abf3104dc9f6d7b491d8ef5194332525d}{bind\+Widget()}} when deployed in Entry\+Point\+Type\+::\+Widget\+Set mode to manage a number of widgets within a 3rd party page.
\item definition of cookies using \mbox{\hyperlink{classWt_1_1WApplication_a5d10106d4131611aa98e20e6a82b3c40}{set\+Cookie()}} to persist information across sessions, which may be read using \mbox{\hyperlink{classWt_1_1WEnvironment_a13e2c7b407afdde4c61eb489cffeee1e}{W\+Environment\+::get\+Cookie()}} in a future session.
\item management of the internal path (that enables browser history and bookmarks) using \mbox{\hyperlink{classWt_1_1WApplication_a2c1a10aadc0d7ed877b5715b42ca4911}{set\+Internal\+Path()}} and related methods.
\item support for server-\/initiated updates with \mbox{\hyperlink{classWt_1_1WApplication_ad9631ca64e68d30d40cb49c90e55223d}{enable\+Updates()}} 
\end{DoxyItemize}

Returns the app\+Root special property. 

This returns the \char`\"{}app\+Root\char`\"{} property, with a trailing slash added to the end if it was not yet present.

The property \char`\"{}app\+Root\char`\"{} was introduced as a generalization of the working directory for the location of files that do not need to be served over http to the client, but are required by the program to run properly. Typically, these are message resource bundles (xml), C\+SV files, database files (e.\+g. S\+Q\+Lite files for \mbox{\hyperlink{namespaceWt_1_1Dbo}{Wt\+::\+Dbo}}), ...

Some connectors do not allow you to control what the current working directory (C\+WD) is set to (fcgi, isapi). Instead of referring to files assuming a sensible C\+WD, it is therefore better to refer to them relative to the application root.

The app\+Root property is special in the sense that it can be set implicitly by the connector (see the connector documentation for more info). If it was not set by the connector, it can be set as a normal property in the configuration file (the default wt\+\_\+config.\+xml describes how to set properties). If the property is not set at all, it is assumed that the app\+Root is C\+WD and this function will return an empty string.

This may be used if you do not want to actively block the current thread while waiting for an event which is needed to complete the current event response. \mbox{\hyperlink{classNote}{Note}} that this effectively freezes the user interface, and thus you should only do this if you know that the event you are waiting for will arrive shortly, or there is really nothing more useful for the user to do than wait for the action to complete.

\char`\"{}\+Resolves\char`\"{} a relative U\+RL taking into account internal paths. 

This resolves the relative U\+RL against the base path of the application, so that it will point to the correct path regardless of the current internal path, e.\+g. if the application is deployed at {\ttfamily http\+://example.com/one} and we\textquotesingle{}re at the internal path {\ttfamily /two/}, so that the full U\+RL is {\ttfamily http\+://example.com/one/two/}, the output of the input U\+RL {\ttfamily three} will point to {\ttfamily http\+://example.com/three}, and not {\ttfamily http\+://example.com/one/two/three}.

If the given url is the empty string, the result will point to the base path of the application.

See the table below for more examples.

\doxyparagraph*{When you would want to use resolve\+Relative\+Url\+:}

Using H\+T\+M\+L5 History A\+PI or in a plain H\+T\+ML session (without ugly internal paths), the internal path is present as a full part of the U\+RL. This has a consequence that relative U\+R\+Ls, if not dealt with, would be resolved against the last \textquotesingle{}folder\textquotesingle{} name of the internal path, rather than against the application deployment path (which is what you probably want).

When using a widgetset mode deployment, or when configuring a base\+U\+RL property in the configuration, this method will make an absolute U\+RL so that the property is fetched from the right server.

Otherwise, this method will fixup a relative U\+RL so that it resolves correctly against the base path of an application. This does not necessarily mean that the U\+RL is resolved into an absolute U\+RL. In fact, Wt will simply prepend a sequence of \char`\"{}../\char`\"{} path elements to correct for the internal path. When passed an absolute U\+RL (i.\+e. starting with \textquotesingle{}/\textquotesingle{}), the url is returned unchanged.

For U\+R\+Ls passed to the Wt A\+PI (and of which the library knows it represents a U\+RL) this method is called internally by the library. But it may be useful for U\+R\+Ls which are set e.\+g. inside a \mbox{\hyperlink{classWt_1_1WTemplate}{W\+Template}}.

Changes the internal path. 

A Wt application may manage multiple virtual paths. The virtual path is appended to the application U\+RL. Depending on the situation, the path is directly appended to the application U\+RL or it is appended using a name anchor (\#).

For example, for an application deployed at\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{http:\textcolor{comment}{//www.mydomain.com/stuff/app.wt}}
\end{DoxyCode}
 for which an {\ttfamily internal\+Path} {\ttfamily \char`\"{}/project/z3cbc/details/\char`\"{}} is set, the two forms for the application U\+RL are\+: 
\begin{DoxyItemize}
\item in an A\+J\+AX session (H\+T\+M\+L5)\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{http:\textcolor{comment}{//www.mydomain.com/stuff/app.wt/project/z3cbc/details/}}
\end{DoxyCode}
 
\item in an A\+J\+AX session (H\+T\+M\+L4)\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{http:\textcolor{comment}{//www.mydomain.com/stuff/app.wt\#/project/z3cbc/details/}}
\end{DoxyCode}
 
\item in a plain H\+T\+ML session\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{http:\textcolor{comment}{//www.mydomain.com/stuff/app.wt/project/z3cbc/details/}}
\end{DoxyCode}
 
\end{DoxyItemize}

\mbox{\hyperlink{classNote}{Note}}, since Wt 3.\+1.\+9, the actual form of the U\+RL no longer affects relative U\+RL resolution, since now Wt includes an H\+T\+ML {\ttfamily meta base} tag which points to the deployment path, regardless of the current internal path. This does break deployments behind a reverse proxy which changes paths.

When the internal path is changed, an entry is added to the browser history. When the user navigates back and forward through this history (using the browser back/forward buttons), an \mbox{\hyperlink{classWt_1_1WApplication_a3e68c4b6bb387f27a614e7962e11967b}{internal\+Path\+Changed()}} event is emitted. You should listen to this signal to switch the application to the corresponding state. When {\ttfamily emit\+Change} is {\ttfamily true}, this signal is also emitted by setting the path (but only if the path is actually changed).

A url that includes the internal path may be obtained using \mbox{\hyperlink{classWt_1_1WApplication_a24cee04e69695ce7f046d20b2473600a}{bookmark\+Url()}}.

The {\ttfamily internal\+Path} must start with a \textquotesingle{}/\textquotesingle{}. In this way, you can still use normal anchors in your H\+T\+ML. Internal path changes initiated in the browser to paths that do not start with a \textquotesingle{}/\textquotesingle{} are ignored.

The {\ttfamily emit\+Change} parameter determines whether calling this method causes the \mbox{\hyperlink{classWt_1_1WApplication_a3e68c4b6bb387f27a614e7962e11967b}{internal\+Path\+Changed()}} signal to be emitted.

Changes the threshold for two-\/phase rendering. 

This changes the threshold for the {\ttfamily size} of a Java\+Script response (in bytes) to render invisible changes in one go. If the bandwidth for rendering the invisible changes exceed the threshold, they will be fetched in a second communication, after the visible changes have been rendered.

The value is a trade-\/off\+: setting it smaller will always use two-\/phase rendering, increasing the total render time but reducing the latency for the visible changes. Setting it too large will increase the latency to render the visible changes, since first also all invisible changes need to be computed and received in the browser. \mbox{\Hypertarget{classWt_1_1WApplication_a6a9a20d65ce9e7c2f62b27387c94e10d}\label{classWt_1_1WApplication_a6a9a20d65ce9e7c2f62b27387c94e10d}}

Handles a browser unload event. 

The browser unloads the application when the user navigates away or when he closes the window or tab.

When {\ttfamily reload-\/is-\/new-\/session} is set to {\ttfamily true}, then the default implementation of this method terminates this session by calling \mbox{\hyperlink{classWt_1_1WApplication_a5231d54ed34982f4366058eb6440c8f7}{quit()}}, otherwise the session is scheduled to expire within seconds (since it may be a refresh).

You may want to reimplement this if you want to keep the application running until it times out.

\begin{DoxyNote}{Note}
There is no guarantee that closing the browser tab sends the unload event. This is because it is at the web browser\textquotesingle{}s discretion whether it still sends requests for a closed tab. It\textquotesingle{}s also possible that there was no connection upon closing the tab. Sessions that don\textquotesingle{}t receive the unload event will eventually time out according to the {\ttfamily session-\/timeout} set in {\ttfamily wt\+\_\+config.\+xml} (this defaults to 10 minutes).
