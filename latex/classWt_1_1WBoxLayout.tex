\doxysubsection{Detailed Description}
A layout manager which arranges widgets horizontally or vertically. 

This layout manager arranges widgets horizontally or vertically inside the parent container.

The space is divided so that each widget is given its preferred size, and remaining space is divided according to stretch factors among widgets. If not all widgets can be given their preferred size (there is not enough room), then widgets are given a smaller size (down to their minimum size). If necessary, the container (or parent layout) of this layout is resized to meet minimum size requirements.

The preferred width or height of a widget is based on its natural size, where it presents its contents without overflowing. \mbox{\hyperlink{classWt_1_1WWidget_a54365f7acdc237895328be54969165f4}{W\+Widget\+::resize()}} or (C\+SS {\ttfamily width}, {\ttfamily height} properties) can be used to adjust the preferred size of a widget.

The minimum width or height of a widget is based on the minimum dimensions of the widget or the nested layout. The default minimum height or width for a widget is 0. It can be specified using \mbox{\hyperlink{classWt_1_1WWidget_a092c7d1ecfde7f004ef120389d91775d}{W\+Widget\+::set\+Minimum\+Size()}} or using C\+SS {\ttfamily min-\/width} or {\ttfamily min-\/height} properties.

You should use \mbox{\hyperlink{classWt_1_1WContainerWidget_af3d9510f569deed060689aa509ecc831}{W\+Container\+Widget\+::set\+Overflow(\+Overflow\+Auto) }} to automatically show scrollbars for widgets inserted in the layout to cope with a size set by the layout manager that is smaller than the preferred size.

When the container of a layout manager does not have a defined size (by having an explicit size, or by being inside a layout manager), or has has only a maximum size set using \mbox{\hyperlink{classWt_1_1WWidget_af023064cf8007bda6baeb5277d79bbd9}{W\+Widget\+::set\+Maximum\+Size()}}, then the size of the container will be based on the preferred size of the contents, up to this maximum size, instead of the default behaviour of constraining the size of the children based on the size of the container. \mbox{\hyperlink{classNote}{Note}} that because of the C\+SS defaults, a W\+Container has by default no height, but inherits the width of its parent widget. The width is thus by default defined.

A layout manager may provide resize handles between items which allow the user to change the automatic layout provided by the layout manager (see \mbox{\hyperlink{classWt_1_1WBoxLayout_ac2628c9ddfb9b6c079a8b5d5261b94c2}{set\+Resizable()}}).

Each item is separated using a constant spacing, which defaults to 6 pixels, and can be changed using \mbox{\hyperlink{classWt_1_1WBoxLayout_a34d544a1a2bd8beffa6b8e5e84fbc19b}{set\+Spacing()}}. In addition, when this layout is a top-\/level layout (i.\+e. is not nested inside another layout), a margin is set around the contents. This margin defaults to 9 pixels, and can be changed using \mbox{\hyperlink{classWt_1_1WLayout_af18bae8c743c9f9a5cabee58826ffb45}{set\+Contents\+Margins()}}. You can add more space between two widgets using \mbox{\hyperlink{classWt_1_1WBoxLayout_a3cbd78251252e724d01937e2f6cc803d}{add\+Spacing()}}.

For each item a stretch factor may be defined, which controls how remaining space is used. Each item is stretched using the stretch factor to fill the remaining space.

\begin{DoxyNote}{Note}
When Java\+Script support is not available, not all functionality of the layout is available. In particular, vertical size management is not available.

When a layout is used on a first page with progressive bootstrap, then the layout will progress only in a limited way to a full Java\+Script-\/based layout. You can thus not rely on it to behave properly for example when dynamically adding or removing widgets. 
\end{DoxyNote}
